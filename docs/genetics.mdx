title: "Genetics"
description: "Upload VCF/WGS or SNP-array data, queue interpretation, and read genetic pipeline results."
---

Genetics runs as a queued job. Upload a file, start analysis, and poll the job.
The hosted worker runs the bundled analyze-health pipeline (ClinVar, CPIC
pharmacogenomics, polygenic risk scores, aging hallmarks, and a Genomic
Longevity Index).

## Choose a provider first

If a user does not yet have genetic data, `GET /wgs-providers` returns a catalog
of WGS, exome, and SNP-array providers with pricing, turnaround, data formats,
raw-data access, and CLIA status. Filter by `type` and `region`.

## Upload and analyze

For VCF, VCF.GZ, SNP-array, 23andMe, or AncestryDNA raw files larger than a few
megabytes, do not base64 encode the file or send it to `/imports/file`. Create
a direct upload session instead. The source is
created as `genetics` with `upload_status: "pending"`; it cannot be analyzed
until it has been finalized.

The hosted private bucket currently accepts individual genetics files up to
512 MB, which includes a 103 MB VCF. Larger files need a storage-plan increase
before creating an upload session.

```bash
# 1. Create a private direct-upload session (the JSON body stays tiny)
SESSION=$(curl -s "$HEALTH_API/genetics/uploads" \
  -H "authorization: Bearer $TOKEN" \
  -H "content-type: application/json" \
  -d '{
    "user_id": "'"$USER_ID"'",
    "organization_id": "'"$ORGANIZATION_ID"'",
    "filename": "genome.vcf.gz",
    "byte_length": 108003328,
    "content_type": "application/gzip"
  }')
UPLOAD_URL=$(jq -r '.upload.url' <<<"$SESSION")
SOURCE_ID=$(jq -r '.source_id' <<<"$SESSION")

# 2. PUT the unmodified file directly to `upload.url`.
# Use exactly the supplied method and headers. This URL is private, scoped to
# one object, and expires after `expires_in_seconds`; it bypasses Cloudflare
# and the API server entirely.
curl --fail-with-body -X PUT "$UPLOAD_URL" \
  -H "content-type: application/gzip" \
  --upload-file genome.vcf.gz

# 3. Confirm that the private storage object is complete.
curl -s -X POST "$HEALTH_API/genetics/uploads/$SOURCE_ID/complete" \
  -H "authorization: Bearer $TOKEN" \
  -H "content-type: application/json" \
  -d '{"user_id":"'"$USER_ID"'","organization_id":"'"$ORGANIZATION_ID"'"}'

# 4. Start analysis (returns a queued job)
curl -s "$HEALTH_API/genetics/analyze" \
  -H "authorization: Bearer $TOKEN" \
  -H "content-type: application/json" \
  -d '{"user_id":"'"$USER_ID"'","organization_id":"'"$ORGANIZATION_ID"'","source_ids":["'"$SOURCE_ID"'"]}'

# 5. Poll the job
curl -s "$HEALTH_API/genetics/jobs/$JOB_ID" -H "authorization: Bearer $TOKEN"
```

Small text genetic exports can still use `POST /imports/file` with
`category: "genetics"`. For an agent using MCP, call `start_genetics_upload`,
PUT the file to the returned `upload.url`, then call
`complete_genetics_upload` with the returned `source_id`. Do not include a
genetics source ID in a biomarker field: a multimodal analysis accepts source
IDs from both modalities separately.

When complete, the analysis gains a `genetic_pipeline_analysis` interpretation
summarizing GLI, matched markers, CPIC matches, and ClinVar findings. Clinical
and medication findings are treated as educational until confirmed by a
qualified clinician. It also gains compact `genetic_consumer_insight`
interpretations. These make supported health, nutrition, sleep, recovery, and
performance topics available through `POST /query` without loading the full
WGS artifact.

Examples include caffeine clearance and sensitivity, aerobic trainability,
power-versus-endurance context, grip strength, fat-free mass, walking duration,
sleep duration, chronotype, exercise tolerance, soft-tissue resilience, and
pulmonary function. Research-only cognitive, personality, and social models
are separately queryable, with direction and population ranking intentionally
withheld. These are probabilistic context, not talent, intelligence,
personality, or physiological tests. A genotype must not override observed
sleep response, training performance, spirometry, laboratory results,
relationships, symptoms, or clinical review.

## Reference modes

- **Compact (default):** bundled ClinVar-derived rsID reference, with no large
  download.
- **Full dbSNP (advanced, hosted GRCh37):** broader rsID recovery. It runs on
  the dedicated Fly WGS worker against a shared encrypted persistent reference
  cache; it is not downloaded inline for an individual request. GRCh38 remains
  a separate setup and is not enabled by this deployment.

### Hosted full-dbSNP status and billing contract

The hosted Fly deployment keeps the 40 GB encrypted `dbsnp_refs` volume mounted
only on an on-demand WGS worker at `/data/reference`. The worker is provisioned
in its own Fly region, independently from the API and wearable worker, and is
started only after a durable genetics job is queued. This keeps routine API and
wearable work available even when WGS capacity is constrained. The worker image
is refreshed to the current API release before each start.

The reference is the official NCBI GRCh37 assembly
`GCF_000001405.25.gz` from the NCBI FTP latest-release directory (downloaded
at the official advertised byte length with its `.md5` and `.tbi` sidecars;
the sidecars remain on the volume for operator verification).
`GET /capabilities` reports `genetics.full_dbsnp` as `available` when the
operator enables the corresponding secrets.

When full dbSNP is provisioned, the reference is shared across jobs - not copied
per user. Plan for roughly 30–40 GB per genome build (GRCh37 and GRCh38 need
separate references), plus temporary space for the input and annotated output.

Full dbSNP must be an explicit, paid hosted capability. Before accepting a
`full_dbsnp` request, the service must verify an active Stripe subscription, a
valid payment method on the Stripe customer, and a plan with a non-zero
`full_dbsnp_jobs` quota (currently intended for Builder, Growth, and Enterprise).
Free and Standard remain compact-only. Self-hosted deployments control their
own reference provisioning and billing policy.

The request contract exposes an explicit annotation choice:
`annotation_depth: "compact" | "full_dbsnp"`. Results preserve the selected
depth and should include reference build/version, annotated count, and
provenance. Do not describe compact annotation as exhaustive genome
interpretation: full dbSNP improves rsID recovery, but it does not make every
variant medically interpretable.

### Consumer score safety contract

Every API-normalized polygenic result has one explicit calculation state:

- `calibrated_absolute_risk`: a validated model returned an absolute risk;
- `reference_relative`: a compatible, named reference panel supports the
  population comparison;
- `raw_score_only`: the weighted sum is available, but direction and percentile
  are withheld;
- `insufficient_coverage`: too few model variants matched for interpretation;
- `research_only`: a sensitive research model was scored, but direction,
  percentile, and individual trait prediction are intentionally withheld;
- `unsupported_model`, `not_applicable`, or `failed_retryable`.

The API never converts an uncalibrated score into "Average" or the 50th
percentile. A reference-relative percentile requires a named calibration
method and reference panel. Results retain score ID and release, source URL,
genome build, matching method, ancestry metadata, model coverage, and
limitations. `dashboard.metadata.consumer_genetics` contains the normalized
section and its `interpretation_release`.

Full-dbSNP analyses additionally run a versioned API-owned position-and-allele
registry after rsID annotation. Registry release `2026-07-19.5` contains 22
pinned PGS Catalog definitions. The performance and recovery expansion adds:

| Consumer topic | PGS Catalog definition | Reporting policy |
| --- | --- | --- |
| Hand grip strength | [PGS001927](https://www.pgscatalog.org/score/PGS001927/) | Consumer context; measurement takes precedence |
| Whole-body fat-free mass | [PGS003901](https://www.pgscatalog.org/score/PGS003901/) | Consumer context; not equivalent to muscle or strength |
| Walking duration | [PGS001073](https://www.pgscatalog.org/score/PGS001073/) | Direction withheld; not motivation or capacity |
| Sleep duration | [PGS001978](https://www.pgscatalog.org/score/PGS001978/) | Consumer context; longer is not automatically better |
| Chronotype | [PGS000336](https://www.pgscatalog.org/score/PGS000336/) | Direction withheld pending model-faithful calibration |
| Fluid-reasoning test research | [PGS001232](https://www.pgscatalog.org/score/PGS001232/) | Research only, non-directional |
| Loneliness research | [PGS001091](https://www.pgscatalog.org/score/PGS001091/) | Research only, non-directional |
| Friendship-satisfaction research | [PGS001398](https://www.pgscatalog.org/score/PGS001398/) | Research only, non-directional |
| Neuroticism research | [PGS002561](https://www.pgscatalog.org/score/PGS002561/) | Research only, non-directional |

The registry also retains [PGS000210](https://www.pgscatalog.org/score/PGS000210/)
for pulmonary function (FEV1/FVC), matched on harmonized GRCh37 coordinates and
alleles. No suitable VO2max or exercise-response PGS was added: those topics
remain direct-marker and measured-phenotype context until an approved,
model-faithful score is available. Score files add about 15 MB uncompressed;
the large HGDP + 1000 Genomes reference panel is separate and is not required
to calculate raw scores.

Research-only models have an enforced `research_only_non_directional`
reporting policy. Even if a compatible reference distribution is installed,
the API returns no percentile, higher/lower label, or prediction of
intelligence, personality identity, social worth, relationship quality,
education potential, or employment potential. These results never drive a
health action or high-impact decision.

A variant-only WGS VCF normally omits homozygous-reference loci. The scorer can
infer those loci only against the configured dbSNP reference under a disclosed
high-coverage WGS assumption. It reports observed calls, inferred-reference
loci, allele mismatches, and missing/uncallable loci separately. This is useful
for a raw model score, but it is not equivalent to all-sites callability and is
one reason calibration remains withheld.

### HGDP + 1000 Genomes reference calibration

Reference-relative results use the PGS Catalog recommended
`pgsc_HGDP+1kGP_v1` panel. A result is upgraded from `raw_score_only` to
`reference_relative` only when all of these gates pass:

- the scoring-file SHA-256, GRCh37 build, scoring algorithm, and weighted
  variant count exactly match the calibration artifact;
- at least 95% of the model variants pass the uploaded-genome matching QC;
- `pgsc_calc` returns a non-low-confidence `MostSimilarPop` assignment from the
  same reference-panel release;
- that super-population has at least 100 unrelated reference samples and a
  valid score distribution.

The API returns the empirical `percentile_MostSimilarPop`, population-relative
z-score, population label, sample count, assignment method, panel release, and
direction status. “Most similar population” is a statistical PCA/classifier
assignment, not identity, ethnicity, nationality, or culture. Population
normalization does not make a score equally predictive across ancestries and
does not convert it to absolute disease risk.

Direction is a separate evidence gate. A percentile can describe where the
model score sits while the trait direction remains withheld. Only models whose
pinned source semantics establish `higher_trait_value` or `higher_liability`
receive higher/lower trait language; an `NR` weight type is never guessed.

Build the compact, de-identified calibration registry from the dedicated
reference-scoring output with:

```bash
npm run prs:calibration:build -- \
  --scores /data/prs/reference-scores.tsv.gz \
  --release 2026-07-19.1 \
  --reference-sha256 <sha256-of-pgsc_HGDP+1kGP_v1.tar.zst> \
  --unrelated-samples <count> \
  --generator-version <pinned-generator-version>
```

The input rows must have been calculated with
`foreverbetter.position_allele.grch37.v1`; mixing the official calculator's
`SUM` with a different matched-variant set is rejected. The generated artifact
contains sorted score distributions only and strips all reference sample IDs.
Configure `HEALTH_ANALYSIS_PGS_CALIBRATION_PATH` when the artifact is stored
outside `data/genetics/pgs/calibration.json`. Until the artifact and a
compatible population-similarity result are present, the valid raw score is
returned with the precise calibration-unavailable reason and reanalysis stays
recommended.

### Query fallback and reanalysis

The WGS worker uses `bcftools` first. If the final annotated VCF is readable but
that rsID query fails, the worker records the original bcftools error and uses a
single-pass text parser so the job can still return interpreted results. Poll
`GET /genetics/jobs/:id` and inspect `result_summary.raw`:

- `annotation_depth_requested` and `annotation_depth_used` identify the selected
  reference mode.
- `rsid_extraction_method` is `bcftools` or `text_fallback`.
- `rsid_extraction_fallback_reason` retains the original query failure when a
  fallback was required.

The job response also exposes `stage`, `progress_pct`, `progress_message`, and
`last_progress_at`, plus `retryable`, `reanalysis_recommended`, and
`reanalysis_reason`. Stable stages include `queued`, `preparing`,
`annotating_variants`, `extracting_genotypes`, `clinical_interpretation`,
`polygenic_scoring`, `consumer_interpretation`, `persisting_results`,
`retry_queued`, `complete`, and `failed`.

A valid VCF with zero rsIDs is not treated as corrupt. The configured compact or
full reference annotation runs before extraction; zero matches in the parser
remain a valid count. If `text_fallback` appears, the current interpreted result
remains available, and a caller can submit a new `/genetics/analyze` request for
the same source after the WGS worker has been repaired. A failed full-dbSNP
annotation itself is still retried by the queue rather than silently relabeled
as a successful full analysis.

Both extraction paths process command output incrementally instead of retaining
the full WGS query in the Node heap. The hosted wrapper streams child output to
worker logs and retains only a bounded diagnostic tail for the persisted error.
The dedicated WGS process is configured with 4 GB of memory and a 3 GB Node heap;
self-hosted deployments should keep similar headroom for `bcftools`, genotype
maps, and temporary files.

After the pipeline renders its result, the worker retries transient PostgreSQL
recovery and connection failures while saving the analysis and terminal job
state. A brief database failover therefore does not discard a multi-hour WGS
result or leave the job silently stuck in `running`. Retry events include the
job ID, operation, attempt, delay, and original database error in worker logs.

### Bounded inline results

The rendered deep-genomic artifact can contain tens of thousands of exploratory
variant cards. The analysis API keeps every clinical-condition, drug-response,
and other-risk card, plus all trait, insight, protocol, PRS, and GWAS summaries.
It does not embed the large low-priority `uncommon_mutations` collection in each
analysis row, and condition-catalog collections are capped at 25 entries per
modality. `dashboard.metadata.persistence_compaction` records the limits and
the exact number omitted from each collection. This keeps `GET /analyses/:id`,
dashboard, action-plan, and query requests usable on the hosted API while
making the bounded result explicit instead of silently dropping data. The
rendered deep-dive artifact remains the appropriate future surface for
paginated exploratory variants.

`GET /analyses/:id/action-plan` reports `status: "processing"`,
`"setup_required"`, or `"failed"` when results are unavailable. These states do
not return a misleading “within target” summary. Only `status: "ready"` means
the empty-plan language can be interpreted as an analyzed result.

Full dbSNP jobs use `HEALTH_ANALYSIS_FULL_DBSNP_TIMEOUT_MS` independently of
the compact analysis timeout. The hosted WGS worker defaults this to four hours
so multi-million-variant annotation is not killed by the 30-minute compact-job
limit.

## Ancestry

`POST /genetics/ancestry` validates source readiness and returns reference-panel
setup requirements. Ancestry proportions are returned once the ancestry worker
(1000 Genomes + PCA/admixture) is configured.
