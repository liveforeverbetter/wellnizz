title: "Design systems"
description: "Curated design-token sets for health & wellness UIs, so you can style your product in minutes."

Building a wellness product's UI is easier when you start from a coherent design
system. `GET /design/systems` returns a curated library of design-token sets
covering color, typography, spacing, radii, elevation, motion, and signature
components. Only the three supported systems are returned: wellnizz,
Aperture, and Meridian. These are **public reference data (no auth required)**.

```bash
curl -s "$FB_API/design/systems" | jq .
curl -s "$FB_API/design/systems/foreverbetter" | jq .
```

## What you get

`GET /design/systems` lists every system with a short `vibe` and what it's
`best_for`. `GET /design/systems/{id}` returns the full token set plus a
ready-to-paste `design_md`:

```json
{
  "id": "aperture",
  "inspired_by": "Aperture Design System handoff",
  "color_scheme": "light",
  "colors": {
    "background": "#F4F6F7",
    "primary": "#0EA5A0",
    "data_viz": ["#0EA5A0", "#5B67EA", "..."]
  },
  "typography": {
    "font_display": "\"Fustat\", \"Segoe UI\", system-ui, sans-serif",
    "google_fonts": [],
    "scale": { "display": { "size": "56px", "weight": 800 } }
  },
  "spacing": { "base_px": 4, "scale_px": [4, 8, 12, 16, 20, 24, 32, 40, 48, 64, 80] },
  "radii": { "md": "14px", "pill": "999px" },
  "components": {
    "aperture_energy_score": { "description": "…", "css": { "font-size": "56px" } }
  },
  "layout": {
    "hero": "aperture-overview",
    "score_word": "Energy",
    "voice": "coach",
    "sections": ["day-brief", "energy-score", "activity-and-sleep"],
    "summary": "A calm, optimistic daily health overview."
  },
  "design_md": "…"
}
```

The `design_md` field is a complete DESIGN.md you can drop into Cursor, Claude
Code, or v0 to steer generated UI toward that look.

Each system also carries a **`layout`**: systems differ in dashboard
structure as well as palette. `hero`, `sections`, and `voice` give the order
and tone, so a client can reproduce the intended information hierarchy instead
of applying a palette-only theme.

## Retrieve the full Aperture, Meridian, or wellnizz handoff

Tokens help an agent design a new UI. If it needs the supplied source material,
use an implementation package. Each handoff includes component source, design
tokens, templates, UI-kit starting points, a component/template manifest,
checksums, and URLs for included binary assets. The wellnizz handoff also
includes a `voice` block (brand tone rules), a `template` block (dashboard
entrypoint and canonical section order), and a `prompt_injection` directive
for reproducible agent-driven dashboard generation.

```bash
curl -s "$FB_API/design/systems/foreverbetter/implementation" > foreverbetter-handoff.json
curl -s "$FB_API/design/systems/aperture/implementation" > aperture-handoff.json
curl -s "$FB_API/design/systems/meridian/implementation" > meridian-dashboard.json
```

Each text file is returned inline with a checksum and a public source URL under
`/design-system-specs/{id}/...`; write it at the supplied path. Download binary
assets from their listed URLs. UI-kit health values are illustrative, not user
records: bind a rendered product to the user's authorized wellnizz data or
show the supplied connection and empty states.

Meridian is a **WHOOP-inspired wearable-performance dashboard**: recovery,
strain, sleep, and connected-data readiness lead the experience. It is an
independent wellnizz implementation, not affiliated with or endorsed by
WHOOP. Its response also contains the pinned production `dashboard/index.html`,
`dashboard/styles.css`, and `dashboard/app.js` package plus the API bindings it
expects. The entry point is `dashboard/index.html` and is intended to be served
at `/dashboard` on the same API origin. Over MCP, call
`get_design_implementation` with `{ "design_id": "foreverbetter" }`,
`{ "design_id": "aperture" }`, or `{ "design_id": "meridian" }`.

Every visual system carries the same full dashboard contract. Alongside tokens
and components it exposes `metrics`, `modality_sections`, `responsive`,
`animations`, `action_plan`, and `data_capture`. The hero, palette, typography,
voice, and section order still change by system, but no visual system drops a
modality or the action plan. These fields let an agent or custom dashboard
builder reproduce the information architecture without copying reference HTML.

`foreverbetter` is the house design: warm-paper editorial with Plus Jakarta Sans
and JetBrains Mono, a single coral accent (`#df1e39`), and a composite Genomic
Longevity Index hero. It reads like a clinical report with editorial heart. Its
implementation response includes the full dashboard template, UI kits for both
the marketing site and the product dashboard, 88 design tokens, 34 component
preview cards, and a `prompt_injection` field agents should use when building
dashboards — directing them to use all available user data, render placeholder
CTAs for missing data, preserve canonical section order, and speak in the
design system's evidence-first voice.

`aperture` is a calm, consumer-health interface with teal-led tokens, Fustat
and Geist Mono type roles, five health-pillar colors, source-aware metric rules,
responsive behavior, action-plan voice, and motion guidance. Retrieve its full
handoff when an agent needs the supplied component source and templates:

```bash
curl -s "$FB_API/design/systems/aperture/implementation" | jq .
```

Its `aperture_energy_score` is deliberately conditional: render it only when a
documented calculation has enough current data; otherwise present the supplied
connection state and the next useful action. The same rule applies to all
derived scores and keeps a generated UI from inventing health values.

## Available systems

| id                | inspired by                | scheme | best for                                                          |
| ----------------- | -------------------------- | ------ | ----------------------------------------------------------------- |
| `foreverbetter`   | wellnizz house design      | light  | full multimodal longevity dossiers, composite healthspan index, editorial reports |
| `meridian`        | WHOOP (Meridian handoff)   | dark   | agent-built wearable apps, recovery, strain, sleep, data readiness |
| `aperture`        | Aperture Design System handoff | light | consumer health coaching, pillars, source-aware daily overview |

## How these are sourced

The API exposes the supplied Aperture and Meridian handoff source: tokens,
components, templates, UI-kit prototypes, manifests, and included assets. It
deliberately excludes each bundle's `uploads/` reference material. Treat the
systems as a starting point and bind the layouts to live, scoped API data rather
than treating the illustrative UI-kit values as health records.
